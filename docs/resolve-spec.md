# ТЗ: Подсекция «resolve» для пропущенных при push задач и команда `tasklean resolve`

## 1. Цель

- Задачи, которые **пропускаются** при `tasklean push` (например, из‑за «changed in UI»), фиксировать в особой **подсекции внутри тела задачи** в `.md` файле.
- Эту подсекцию **не перезаписывать** при `tasklean pull` (то есть при подтягивании данных с трекера сохранять её и сливать с актуальным содержимым задачи).
- Команда **`tasklean resolve`** выводит список задач, у которых есть эта подсекция — то есть задачи, которые нужно разрешить вручную (решить конфликт локальной и удалённой версии).

## 2. Текущее поведение (без изменений)

- **push:** для каждой задачи, существующей на remote, сравниваются `remote.UpdatedAt` и локальный `UpdatedAt` (из `_index.json`). Если на трекере задача новее — push её **пропускает** (skip), в лог пишется `skip … (changed in UI)`. В файлы задач при этом ничего не записывается.
- **pull:** получает задачи с трекера и для каждой вызывает `storage.SaveTask(task, directory)`. Тело задачи формируется как `## Title + Description` и **полностью перезаписывает** соответствующий блок в `.md` файле. Любые локальные дополнения в теле задачи теряются.

## 3. Формат подсекции «resolve» в теле задачи

### 3.1. Расположение и разметка

- Подсекция находится **внутри** блока одной задачи (между маркером `[📋 id:...]` и разделителем `---` или концом файла).
- Используется фиксированный заголовок третьего уровня, чтобы однозначно распознавать блок и не путать с обычным контентом пользователя.

**Вариант разметки (рекомендуемый):**

```markdown
[📋 id:abc-123]

## Заголовок задачи

Обычное описание задачи...

### tasklean: resolve

Пропущено при push: 2025-03-03T14:00:00Z — задача изменена в UI (remote новее). Разрешите вручную: обновите локально через pull и при необходимости объедините изменения, либо отредактируйте в UI и снова push.
```

- Заголовок подсекции: **`### tasklean: resolve`** (ровно такая строка, без пробелов в начале для уровня H3).
- Всё от следующей строки после этого заголовка до следующего заголовка того же или высшего уровня (`##` или `###`) либо до конца блока задачи считается **телом блока resolve**. Если заголовков нет — до конца блока (до `---` или конца файла).

### 3.2. Правила

- При **парсинге** задачи: если в `Body` блока есть подстрока `### tasklean: resolve`, выделить и сохранять «resolve-блок» (заголовок + тело до границы блока задачи).
- При **записи** задачи из pull: если в текущем локальном блоке этой задачи был resolve-блок — при формировании нового тела задачи **добавить его в конец** (после описания с remote). Таким образом, `tasklean pull` **не перезаписывает** подсекцию resolve, а сохраняет её.
- При **записи** из push (см. ниже): при добавлении/обновлении подсекции использовать тот же формат.

## 4. Изменения в `tasklean push`

- В месте, где принимается решение **пропустить** задачу (например, из‑за `remote.UpdatedAt.After(localSec)`):
  - Не только вывести в консоль `skip … (changed in UI)`.
  - Вызвать новую функцию хранилища/команды, которая:
    - Находит файл и блок с этой задачей (по ID).
    - Читает текущий body блока.
    - Добавляет или **заменяет** подсекцию `### tasklean: resolve` актуальным текстом (дата/время skip, краткая причина, рекомендация разрешить вручную).
    - Сохраняет блок обратно в тот же файл (остальные блоки и структура файла не меняются).
- Если подсекция уже есть — обновить её содержимое (одна запись на задачу: последний skip с новой датой/причиной).

Детали реализации (где именно вызывать — из command или из storage, как передавать причину skip) оставить на этап кода; в ТЗ фиксируем необходимость «при skip писать resolve-блок в тело задачи в .md».

## 5. Изменения в `tasklean pull`

- При сохранении задачи, полученной с remote (`storage.SaveTask` или эквивалент):
  - Перед записью в файл проверить, есть ли в директории уже файл с блоком этой задачи (по ID).
  - Если есть — прочитать текущий body этого блока и выделить из него resolve-блок (по маркеру `### tasklean: resolve`).
  - Если resolve-блок найден — при формировании тела для записи **добавить его в конец** (после `## Title` и описания, пришедших с remote).
  - Записать объединённое тело в блок задачи как обычно.

В результате подсекция «tasklean: resolve» при pull **не перезаписывается** и остаётся видимой пользователю для ручного разрешения.

## 6. Команда `tasklean resolve`

- **Назначение:** вывести список задач, у которых в текущем дереве задач (для указанного remote или директории) есть подсекция `### tasklean: resolve`.
- **Вызов (предлагаемый):**  
  `tasklean resolve [name]`  
  где `name` — имя remote (как у pull/push). Директория берётся из конфига remote. Либо без аргумента — использовать default remote / текущую директорию (уточнить при реализации по аналогии с pull/push).
- **Вывод:** по одной строке (или краткому блоку) на задачу: идентификатор задачи, заголовок, путь к файлу, при необходимости — первый абзац тела resolve (или дата skip). Цель — пользователь сразу видит, какие задачи требуют ручного разрешения.
- **Без подсекций:** если ни у одной задачи нет resolve-блока — вывести сообщение вроде «No tasks with resolve block» и exit 0.

Дополнительные флаги (например, `--clear` для удаления блока после разрешения) можно ввести позже; в базовом ТЗ достаточно вывода списка.

## 7. Хранение и границы

- **Только в .md:** подсекция хранится только в теле задачи в markdown-файлах. В `_index.json` отдельно хранить resolve-статус не требуется (достаточно наличия блока в тексте).
- **Один блок на задачу:** в одном блоке задачи — не более одной подсекции `### tasklean: resolve`. При повторном skip при push — обновлять текст этой подсекции.
- **pull не трогает resolve:** при pull перезаписываются только заголовок и описание из remote; resolve-блок всегда сохраняется и дописывается в конец тела.

## 8. Сценарии использования

1. Пользователь правит задачу в UI Plane. Локально та же задача уже была изменена. При `tasklean push` задача пропускается, в её блок в .md добавляется подсекция «tasklean: resolve» с датой и причиной.
2. Пользователь запускает `tasklean resolve` — видит список таких задач и может открыть файлы, решить конфликт (обновить локально через pull, слить изменения, или оставить версию UI и обновить индекс).
3. После разрешения пользователь вручную удаляет подсекцию `### tasklean: resolve` из .md (или в будущем — команда `resolve --clear <id>`). При следующем `tasklean resolve` эта задача уже не показывается.
4. При следующем `tasklean pull` подсекция resolve у тех задач, где она ещё есть, сохраняется и не затирается.

## 9. Краткий чеклист реализации

- [ ] В storage (или отдельном пакете): парсинг body с выделением resolve-блока; функция слияния body от remote + сохранённый resolve-блок.
- [ ] В storage: функция «добавить/обновить resolve-блок в задаче по ID» (поиск файла, парсинг блоков, обновление body, запись).
- [ ] push: при skip вызывать запись resolve-блока с датой и причиной.
- [ ] pull: при SaveTask подставлять сохранённый resolve-блок в body перед записью.
- [ ] Команда `resolve`: обход задач в директории remote, вывод списка задач с resolve-блоком (ID, заголовок, путь к файлу).
- [ ] Документация и тесты (парсинг/merge resolve, команда resolve).
